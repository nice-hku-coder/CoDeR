## 1. 整体执行链路

当前 `CoDeR` 的实验链路可以压缩成下面这条主线：

```text
PoC
  -> build_triplets.py
  -> encoder_trainer/train_constraint_encoder.py
  -> eval_constraint_encoder.py
  -> build_constraint_benchmark.py
  -> rag_grid_search.py / rag_category_report.py
  -> build_retrieval_benchmark.py
  -> eval_retrieval_metrics.py
  -> run_bm25_baseline.py / run_cross_encoder_baseline.py
  -> rag_e2e_eval.py / latency_benchmark.py / error_analysis_report.py
```

如果按“研究问题”来分，这条链路其实对应 4 个问题：

1. **问题是否存在？**
   - `poc_negation_gap.py`
2. **Constraint Encoder 是否学到了东西？**
   - `encoder_trainer/train_constraint_encoder.py`
   - `eval_constraint_encoder.py`
3. **Dual-view retrieval 是否提升了约束遵循？**
   - `rag_eval.py`
   - `rag_grid_search.py`
   - `rag_category_report.py`
4. **在正式 benchmark 上，它是否优于 baseline，且成本如何？**
   - `eval_retrieval_metrics.py`
   - `run_bm25_baseline.py`
   - `run_cross_encoder_baseline.py`
   - `rag_e2e_eval.py`
   - `latency_benchmark.py`

### 2.1 基建脚本

## `experiments/common.py`

### 作用
整个项目的基础工具箱。

### 核心内容

- 项目路径常量：
  - `DATA_DIR`
  - `PROCESSED_DIR`
  - `OUTPUTS_DIR`
  - `REPORTS_DIR`
  - `CHECKPOINTS_DIR`
- `ensure_project_dirs()`：保证目录存在
- `set_seed()`：固定随机种子
- `read_jsonl()` / `write_jsonl()`
- `pairwise_accuracy()`：用于 triplet 离线评估
- `load_sentence_encoder()`：加载 sentence-transformers 模型，并自动修复本地 causal LM 缺少 `pad_token` 的兼容问题

### 为什么重要

这是很多脚本都依赖的底层公共模块；
尤其 `load_sentence_encoder()` 很关键，因为本地 Llama 主题模型能跑通，很大程度靠这个兼容逻辑。

---

## `experiments/metrics.py`

### 作用
统一定义检索指标。

### 提供的指标
  - `recall_at_k`:在应该被找回来的文档里，前 k 个结果找回来了多少
  - `precision_at_k`:top-k 里有多少是想要的正样本
  - `ccr_at_k`:top-k 检索结果中，有多少比例是满足 query 约束的
  - `ndcg_at_k`:看相关文档是否被排到了靠前位置

### 你要特别记住

`CCR@k` 在代码里本质上就是：

> top-k 上对 `constraint_satisfying_doc_ids` 做 precision。

即：

```text
CCR = top-k 里满足约束的文档占比
```

### 2.2 问题证明 / 动机脚本

## `experiments/poc_negation_gap.py`

### 作用
证明普通 embedding 在 contradiction / negation 场景里确实有混淆。

### 输入

- HuggingFace `snli`
- split: 默认 `validation`

### 做法

- 收集 `(premise, entailment, contradiction)` 对；
- 分别编码：
  - premise
  - entailment
  - contradiction
- 计算：
  - `sim(premise, entailment)`
  - `sim(premise, contradiction)`
  - margin = 前者 - 后者

### 输出

- `outputs/reports/poc_negation_gap/poc_negation_gap_summary.txt`
- `outputs/reports/poc_negation_gap/poc_negation_gap_scores.csv`
- `outputs/figures/poc_negation_gap_hist.png`。

### 2.3 数据构建脚本

## `experiments/build_triplets.py`

### 作用
生成训练约束编码器的 triplet 数据，并顺手生成小规模 smoke 数据。

### 输入

- `snli` 数据集

### 输出

- `train_triplets.jsonl`
- `val_triplets.jsonl`
- `smoke_eval.jsonl`
- `demo_corpus.jsonl`

### triplet 格式

```json
{
  "query": "...",
  "positive": "...",
  "hard_negative": "..."
}
```

### 关键逻辑

- 同一个 `premise` 下，如果同时出现：
  - label 0: entailment
  - label 2: contradiction
- 就构造一条：
  - `query = premise`
  - `positive = entailment`
  - `hard_negative = contradiction`

---

## `experiments/build_constraint_benchmark.py`

### 作用
构造大规模约束评测集。

### 三个类别

- `negation`
- `exclusion`
- `numeric`

### 数据特点

- 完全模板化硬编码合成
- 每条 query 对应 6 条左右 docs
- 每个 doc 都有 `satisfies: 0/1`

### 输出

- `constraint_benchmark_v1.jsonl`

### 风险

- 太模板化，真实分布覆盖有限
- 但优点是可控、可快速迭代

### 输入

- `retrieval_benchmark_v1.jsonl`

### 输出结构

### 2.4 训练与离线验证脚本

## `encoder_trainer/train_constraint_encoder.py`

### 作用
训练 Encoder B。

### 输入

- `train_triplets.jsonl`

格式：
```text
(query, positive, hard_negative)
```

目标是：

- `sim(query, positive)` 尽可能大
- `sim(query, hard_negative)` 尽可能小

### 模型结构

- `Transformer(base_model)`
- `Pooling`
- `SentenceTransformer`

### 默认参数

- base model: `distilbert-base-uncased`
- epochs: `1`
- batch-size: `16`
- max-seq-len: `128`

### 损失函数

- `MultipleNegativesRankingLoss`

### 输出

- `outputs/checkpoints/constraint-encoder-v1/`

---

### 指标

## `experiments/rag_grid_search.py`

### 作用
搜索最优 `alpha / tau`。

### 输入


### 核心逻辑

- 先预计算每个 query 上：
  - topic scores
  - constraint scores
- 遍历参数网格：
  - `alpha`
  - `tau`
- 对每组参数计算平均 CCR
- 若 `tau` 过滤后为空，则回退到 topic rank

### 输出

- `outputs/reports/rag_grid_search/*.json`

### 为什么重要

- 确定实验主配置；
- 当前最佳点多次显示为：`alpha=0.0, tau=0.6`。

### 2.6 正式评测与 baseline 脚本

## `experiments/eval_retrieval_metrics.py`

### 作用
这是当前最核心的主评测脚本。

### 输入

- `retrieval_benchmark_v1.jsonl`
- `retrieval_corpus_v1.jsonl`
- topic model
- constraint model
- `retrieve-k`
- `alpha`
- `tau`

### 输出指标

- `Recall@10`
- `Recall@100`
- `NDCG@10`
- `CCR@10`

### 输出粒度

- overall
- by_category
- per_query

### dual ranking 的关键逻辑

1. 先用 topic score 得到全库 vanilla 排序；
2. 取前 `retrieve_k` 作为 dual 候选池；
3. 在候选池上算融合分；
4. 用 `tau` 按 constraint score 过滤；
5. 若有保留候选，则在保留候选里按融合分排序；
6. 若没有候选通过 `tau`，则回退到 vanilla 候选池顺序；
7. 最后把剩余文档按 vanilla 顺序拼回去，形成完整排名。

## `experiments/run_cross_encoder_baseline.py`

### 作用
提供更强的 reranker baseline。

### 输入
- `retrieval_benchmark_v1.jsonl`
- `retrieval_corpus_v1.jsonl`

### 做法

1. 先用 topic model 召回前 `candidate-k`；
2. 构造 `(query, doc)` 对；
3. 用 CrossEncoder 打分；
4. 对候选重排；
5. 再把剩余文档按 topic 顺序拼回去。

### 兼容细节

脚本兼容 cross-encoder 输出二维 logits 的情况：

- 会尝试从 `id2label` 中找 entailment 维度
- 找不到时默认最后一维

### 输出

- `outputs/reports/baselines/cross_encoder_metrics_report.json`

### 当前边界

当前用的 cross-encoder 偏 NLI，不是最强 IR reranker。

### 2.7 端到端代理评测 / 分析脚本

## `experiments/rag_e2e_eval.py`

### 作用
基于 retrieval report 做 proxy 端到端评估。

### 核心规则

如果 top1 满足约束，则认为回答 faithful。

### 输出

- `proxy_faithfulness`
- `negation_failure_rate`
- 各类别 proxy faithfulness

### 注意

这是 proxy，不是真正的 answer-level faithfulness。

### 方案

- vanilla

- 可选 cross-encoder

### 输出

- `mean_ms`

## `experiments/error_analysis_report.py`

### 作用
自动生成错误案例 markdown。

### 输入

- retrieval report
- corpus file

### 做法

- 读取 `per_query`
- 比较 vanilla / dual 的 `ccr@10`
- 检查 dual top1 是否满足约束
- 用规则给失败打标签：
  - `negation-scope failure`
  - `numeric-threshold failure`
  - `exclusion-conflict not filtered`
  - `dual worse than vanilla`
  - `hard residual case`

### 输出

- `outputs/reports/latency_and_error/error_analysis_cases.md`

### 意义

这份报告很适合拿去写论文的 error analysis 小节。

## 6. 你和学长对齐时最值得问的 8 个问题

后续你很可能要补实验或写论文，建议直接问学长下面这些问题：

1. 当前论文主表是继续用 `CCR@3`，还是切到 `@10` 多指标主表？
2. 当前 Local Llama 是不是确定作为主 topic backbone？
3. Cross-Encoder baseline 是否准备换成更强 IR reranker？
4. `constraint_benchmark_v1` 是否只作为开发集，是否还要加真实 query 测试集？
5. Negation 类是否准备单独做数据增强？
6. `rag_e2e_eval.py` 的 proxy faithfulness 是否只做辅助结果，还是想升级成正式 e2e？
7. 当前结果里最想打的核心 claim 是“CCR 提升”，还是“整体 retrieval quality 也提升”？
