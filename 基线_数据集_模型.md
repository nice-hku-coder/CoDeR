# 基线、数据集与更强 baseline 调查

## 1. 目的：回应当前两个核心问题：
- `constraint_benchmark_v1.jsonl / retrieval_corpus_v1.jsonl / retrieval_benchmark_v1.jsonl` 主要靠模板合成，自主生成痕迹太重；
- 当前 baseline 偏弱，论文说服力不足。

---

## 2. BM25

### 2.1 价值

BM25 是必须保留的下界基线：
- **BM25**：最基础 lexical baseline

### 2.2 对 CoDeR 的意义

如果 CoDeR 只赢纯 BM25，审稿人会说：
- “你只是赢了最朴素的 lexical baseline”

如果 CoDeR 能稳定优于纯 BM25，则至少能说明：
- 方法不是只在 dense baseline 上有效
- 相比最基础 lexical retrieval，constraint-aware retrieval 确有额外价值

### 2.3 实施难度

低。当前已有 BM25 脚本与评测流程，可直接保留为 lexical 下界基线。

### 2.4 建议优先级

**高**。这是必须保留的下界基线。

---

## 3. Contriever

### 3.1 文献结论

`Contriever` 是经典 zero-shot unsupervised dense retriever。

论文要点：
- 用无监督对比学习训练 dense retriever
- 在 BEIR 上 Recall@100 常可与 BM25 竞争
- 作为 zero-shot dense baseline，社区认可度高

### 3.2 对 CoDeR 的意义

它可以回答一个非常关键的问题：

> CoDeR 的提升，到底是因为“任意一个更强 dense encoder 都会更好”，还是因为我们真的建模了 constraint compliance？

如果 CoDeR 只比 MiniLM baseline 强，审稿人会怀疑 topic encoder 太弱。
但如果能和 Contriever 对比，就更容易说明：
- 即使换成成熟的 zero-shot dense retriever，constraint violation 仍然存在
- CoDeR 的 gain 不是单纯 backbone gain

### 3.3 适配方式

可做两种版本：

#### 版本 A：Vanilla Contriever
- 用 Contriever 直接编码 query/doc
- 按 topic relevance 排序
- 在同一 benchmark 上评估 Recall@k / NDCG@k / CCR@k

#### 版本 B：Contriever + CoDeR constraint rerank
- 把 Contriever 当 topic backbone
- 检验 CoDeR 是否对不同 topic encoder 都稳定

### 3.4 建议优先级

**高**。这是最自然的强 zero-shot dense baseline 之一。

---

## 4. BGE

### 4.1 文献结论

`BGE` 不是“某个特殊任务方法”，而是一类非常强的通用 embedding backbone。

C-Pack / BGE 论文强调：
- BGE 在通用 embedding、检索、重排、STS 上表现很强
- BGE 有不同尺寸可选，易部署
- 已被 RAG/embedding 社区广泛采用

### 4.2 对 CoDeR 的意义

BGE 的最大价值不是“论文故事”，而是**实证说服力**：
- 如果你当前 topic backbone 是 MiniLM 或本地 llama embedding，审稿人可能觉得 backbone 不够强
- 若 CoDeR 在 BGE backbone 上仍能提升 CCR / proxy faithfulness，就很有说服力

### 4.3 推荐用法

建议至少补两档：
- **BGE-base / bge-large-en-v1.5 vanilla retriever**
- **BGE + CoDeR dual-view rerank**

如果资源有限，先跑：
- bge-large-en-v1.5 作为 strongest off-the-shelf dense retriever

### 4.4 建议优先级

**非常高**。从“最容易让审稿人信服”的角度，BGE 很重要。

---

## 5. HyDE

### 5.1 文献结论

`HyDE`（Hypothetical Document Embeddings）是经典 LLM-enhanced zero-shot retrieval 方法。

核心做法：
1. 让 LLM 根据 query 生成一个 hypothetical document
2. 用 Contriever 等 encoder 编码该伪文档
3. 用这个向量去检索真实文档

论文显示：
- 它显著优于纯 Contriever
- 在 TREC DL / BEIR 上是强 zero-shot baseline

### 5.2 对 CoDeR 的意义

HyDE 特别适合作为“你们不是唯一用 LLM/生成增强 retrieval 的方法”的对照。

如果 CoDeR 能优于 HyDE，说明：
- 问题不只是 query expansion 或 pseudo-document generation 能解决
- 对 constraint 的显式建模比纯生成式 query reformulation 更有效

### 5.3 风险

HyDE 主要增强 topic relevance，不一定天然擅长：
- negation scope
- exclusion filtering
- numeric threshold consistency

但这正好是它作为 baseline 的价值所在：
- 它可能提高 Recall / NDCG
- 未必提高 CCR
- 因而能突出 CoDeR 的独特贡献

### 5.4 建议优先级

**中高**。如果时间只够补 2~3 个强 baseline，HyDE 是很好的第三/第四选择。

---

## 6. DPR

### 6.1 文献结论

`DPR` 是最经典的 supervised dense retriever 之一。

核心特点：
- 双塔结构
- in-batch negatives
- 用标注 query-passage 对训练

### 6.2 对 CoDeR 的意义

DPR 的价值不在“zero-shot公平”，而在**参考上界和主流强基线**：
- 审稿人熟悉
- Dense retrieval 经典方法
- 如果 CoDeR 连 DPR 系列都不对比，会显得 baseline 缺口明显

### 6.3 注意事项

DPR 原生是 QA passage retrieval，不是专门为 negative constraints 设计。
所以更合理的解读不是：
- “DPR 输了，所以 dense retriever 都不行”

而是：
- “即便是经典 supervised dense retriever，也不天然具备约束遵循能力”

### 6.4 建议优先级

**中高**。因为认知度高，适合进入主表或附录表。

---

## 7. ANCE

### 7.1 文献结论

`ANCE` 用全局 hard negative mining 强化 dense retrieval。

核心特点：
- 用 ANN index 从全库挖 hard negatives
- 比普通 in-batch negatives 更强
- 在 web search / OpenQA 中曾是强 dense retrieval baseline

### 7.2 对 CoDeR 的意义

ANCE 比 DPR 更“强硬”的地方在于：
- 它专门强化 difficult negatives 的区分

这和 CoDeR 的问题设定非常接近，因为 constraint-violating docs 本质上也是一类 hard negatives。

因此 ANCE 很适合做如下对照：

> 普通 hard negative dense retrieval vs explicit constraint-aware retrieval

如果 CoDeR 能优于 ANCE，说明：
- 不是“更强 hard-negative dense training”就足够
- constraint 需要单独建模，而不仅是更强 general retriever

### 7.3 建议优先级

**高**。从方法相关性上，它比 DPR 更适合作为 CoDeR 的强 baseline。

---

## 8. NS-IR（最相关）

### 8.1 文献结论

`NS-IR`（2025）几乎是当前最接近 CoDeR 任务定义的工作。

论文核心：
- 目标就是 negative-constraint queries
- 提出 neuro-symbolic retrieval
- 用 LLM 把 query/doc 翻译成 FOL
- 通过 logic alignment + connective constraint 重排候选

### 8.2 为什么它非常关键

因为它能直接堵住一个潜在审稿意见：

> 你们做“constraint-aware retrieval”并不是唯一工作，有没有和更直接处理逻辑约束的模型比较？

NS-IR 就是这样一个最相关 baseline。

### 8.3 对 CoDeR 的实际意义

- 它和 CoDeR 一样，明确把“逻辑一致性/约束一致性”放到检索阶段处理
- 它不是泛化型 dense retriever，而是专门面向 negative-constraint query 的方法
- 它提供了一个很自然的 related-work / strongest conceptual baseline

### 8.4 建议优先级

**中高**。它很值得作为“最相关方法 baseline / related work”保留，但不再建议把它绑定为数据集方案核心。

## 9. 可作为真实语料来源，重构 constraint benchmark 的数据集

这类数据集非常适合当语料库或正例来源。

### 9.1 Wikipedia dump

它也是构造 negative / exclusion 类型 benchmark 的自然语料来源。

#### 价值
- 真实文档分布
- 覆盖广
- 易于构造 entity/topic + exclusion constraint

#### 适合构造的约束
- negation / exclusion
- 属性型筛选
- 某实体相关但不包含某子主题

#### 适配方式
可参考 NS-IR 这类工作的方法思想，但自行重建数据：
1. 从 Wikipedia passage 中选 topic A 的正例文档
2. 找包含被排除实体/子主题 a 的干扰文档
3. 用 LLM 改写成多样 query
4. 人工/规则校验

#### 结论
**非常适合作为 CoDeR v2 benchmark 的真实语料底座。**

---

### 9.2 BEIR 子集

`HyDE / InteR / Contriever` 都大量使用 BEIR。

其中更值得关注的子集：
- SciFact（事实验证）
- FiQA（金融问答）
- DBPedia（实体检索）
- ArguAna（长 query / 对立观点）
- NFCorpus（医学）

#### 为什么适合
这些数据集不直接提供 negation label，但它们提供：
- 真实 query 与真实 corpus
- 多领域场景
- 可二次加工成 constraint benchmark

#### 可能的二次构造方式
- 在 SciFact / FEVER 风格数据中，把 contradiction / refute 作为 constraint-violating doc（**适合增强 negation 类评测与训练。**）
- 在 DBPedia / entity retrieval 中构造 “关于 A 但不涉及 a” 的 exclusion query
- 在 FiQA / product/finance 场景中构造 numeric threshold query

#### 结论
**适合做“真实分布语料 + 二次标注”路线。**

---

## 10. 对 CoDeR 最实用的数据集方案建议

这里给出三档方案。

### 10.1 最小可落地方案

#### 方案内容
保留现有 synthetic benchmark，但尽快引入**真实语料来源**：
- 用 Wikipedia passages 重做 negation / exclusion 子集
- numeric 类暂时保留 synthetic 版本

#### 意义
- 解决“全是自己生成”的质疑
- 不依赖未开源外部数据集
- 可以先把 benchmark 区分为：
  - synthetic dev benchmark
  - real-corpus test benchmark

#### 结论
**这是当前最稳妥、最值得立刻做的事情。**

---

### 10.2 中等投入方案

#### 方案内容
构建 `constraint_benchmark_v2`：
- 保留 synthetic numeric 子集
- 用 Wikipedia / BEIR / FEVER / SciFact 衍生样本补充 negation 与 exclusion 子集
- corpus 从真实 passages 构建

#### 结果形式
最终形成：
- `constraint_benchmark_v2.jsonl`
- `retrieval_corpus_v2.jsonl`
- `retrieval_benchmark_v2.jsonl`

其中：
- negation/exclusion：真实语料 + 自行构造/人工校验
- numeric：保留当前自建，但建议增加真实商品/价格文本来源

#### 结论
**这是最适合论文主实验的新版本。**

---

### 10.3 高投入方案

#### 方案内容
基于 Wikipedia / BEIR / FEVER / SciFact 重新做一套人工校验 benchmark：
- negation：来自 FEVER/SciFact/refute 风格样本
- exclusion：来自 Wikipedia 实体子主题排除
- numeric：来自真实商品/金融/医疗数据
- 每条 query 配 topical positives / constraint-violating hard negatives / irrelevant negatives

#### 结论
最好，但工作量最大，不适合短周期补实验。

## 12. 模型选择建议

### 12.1 建议采用的具体模型

#### Group A：Lexical baselines
- **BM25**：继续使用现有 BM25 实现即可

#### Group B：General dense retrievers
- **Contriever（论文原始选择）**：`facebook/contriever`
- **BGE**：`BAAI/bge-large-en-v1.5`
- **DPR（论文原始选择）**：
  - query encoder：`facebook/dpr-question_encoder-single-nq-base`
  - doc encoder：`facebook/dpr-ctx_encoder-single-nq-base`
- **ANCE**：
  - **paper-faithful**：优先 ANCE 官方代码，采用 **RoBERTa-base Siamese** 路线，并在 OpenQA 设定下从 **DPR checkpoint warm-start**
  - **practical HF 近似**：`sentence-transformers/msmarco-roberta-base-ance-firstp`

#### Group C：LLM-enhanced retrievers
- **HyDE（论文原始选择）**：`text-davinci-003` + `facebook/contriever`
- **HyDE（开源替代）**：`Qwen/Qwen2.5-7B-Instruct` + `facebook/contriever`

#### Group D：Constraint / logic-aware retrievers
- **NS-IR**：`gpt-4o`

#### Group E：Our method
- **CoDeR topic backbone**：`BAAI/bge-large-en-v1.5`

---

### 12.2 各模型与原论文的对应关系

#### 12.2.1 Contriever：`facebook/contriever`
这项选择与原论文是对齐的。

#### 12.2.2 DPR：`facebook/dpr-question_encoder-single-nq-base` + `facebook/dpr-ctx_encoder-single-nq-base`
这项选择与原论文是对齐的。

#### 12.2.3 ANCE：优先保持与论文训练路线一致
ANCE 是当前最需要说明清楚的一个，因为它和 DPR 不同，**关键不只是模型名，而是训练机制**。

根据 ANCE 论文：
- 主体是 **BERT-Siamese / Dual Encoder** dense retriever；
- 在 TREC DL 实验里，作者说明所有 DR 模型都从 **RoBERTa-base** 初始化；
- 并在 `[CLS]` 上接一个 **768×768 projection layer + layer norm**；
- 核心贡献不是某个固定 checkpoint，而是：
  - 用 ANN index 从全库动态挖 hard negatives；
  - 异步刷新索引；
  - 用这些 global hard negatives 继续训练。
- 在 OpenQA 实验里，论文明确说 **ANCE 用 released DPR checkpoints 做 warm-up**。

因此，对 CoDeR 来说应区分两种实现目标：

**方案 A：paper-faithful**
- 不把 ANCE 简化成“随便一个叫 ance 的 checkpoint”；
- 而是说明：
  - backbone 走 **RoBERTa-base Siamese**；
  - 若做 OpenQA 风格设定，可用 **DPR checkpoint warm-start**；
  - 最关键是保留 **ANN hard negative mining** 训练流程。

**方案 B：practical HF**
- `sentence-transformers/msmarco-roberta-base-ance-firstp`
- 但文中要明确写清：
  - 这是 **ANCE 风格的近似可复现实现**；
  - 不是完全复刻论文的 OpenQA / ANN 异步训练设定。

#### 12.2.4 HyDE：
根据 HyDE 论文：
- HyDE 不是单独训练一个新 retriever；
- 它直接把现成 instruction-following LLM 和对比学习 encoder 串起来；
- 原论文明确写的是：
  - 生成模型使用 **InstructGPT / `text-davinci-003`**；
  - 英文 retrieval encoder 使用 **Contriever**；
  - 多语言用 **mContriever**。

因此，最 paper-faithful 的 HyDE 选择应写成：
- **HyDE= `text-davinci-003` + `facebook/contriever`**

替代版本：
- **HyDE= `Qwen/Qwen2.5-7B-Instruct` + `facebook/contriever`**
