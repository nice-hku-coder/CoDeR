# 项目总览与实验现状

## 1. 理解项目

这个项目在做的是：

**Constraint-Aware RAG / Dual-View Retrieval**。

核心目标不是单纯提升“语义相关性”，而是解决 RAG 检索里一种很常见但很隐蔽的错误：

> 检索到的文档虽然和 query 主题相关，但**违反了 query 里的显式约束**。

典型例子：

- `Find hotels that are not dirty`
- `Meals without peanuts`
- `Phones under 100 dollars`

普通 dense retriever 往往能抓到“hotel / peanuts / phone”这些主题词，却分不清：

- `not dirty` vs `dirty`
- `without peanuts` vs `contains peanuts`
- `under $100` vs `$699`

因此项目提出一个**双视角检索框架**：

- **Encoder A / Topic Encoder**：负责主题召回（topic relevance）
- **Encoder B / Constraint Encoder**：负责约束一致性（constraint compliance）

在线流程是：

1. 用 Topic Encoder 先召回候选；
2. 用 Constraint Encoder 对候选做打分；
3. 再进行过滤或重排；
4. 最终把更“相关且满足约束”的文档送给生成模型。

---

## 2. 论文问题定义与方法主张

### 2.1 论文试图回答的问题

1. **不替换 base retriever 的前提下，能否改善检索的逻辑约束遵循？**
2. **哪类约束最受益于专门的 constraint encoder？**
3. **不同 topic backbone 下，这个行为是否稳定？**

### 2.2 方法核心

项目采用的是一个轻量双视角结构：

- **Encoder A (Topic Expert)**：负责召回
- **Encoder B (Constraint Expert)**：负责判断 query-doc 是否满足约束

候选文档的最终分数：

```text
Score(d) = alpha * Sim_topic(q, d) + (1 - alpha) * Sim_constraint(q, d)
```

此外还有一个硬阈值：

```text
keep(d) = 1[ Sim_constraint(q, d) >= tau ]
```

即：

- `alpha` 控制“偏主题”还是“偏约束”；
- `tau` 控制约束过滤严格程度；
- 如果所有候选都被 `tau` 过滤掉，则回退到 topic 排序。

### 2.3 当前方法的真实工程形态

从代码看，这套方法不是 end-to-end jointly trained RAG，而是一个**模块化、插拔式检索增强组件**：

- 不替换现有 topic retriever；
- 在 retriever 和 generator 中间插入 constraint encoder；
- 所以工程上更像“轻量检索防线”。

这点也是论文里一个很重要的 selling point：

> 比 cross-encoder reranking 更轻、更容易接入。

## 3. 当前 baseline 和实验框架

### 3.1 当前已有 baseline

#### Baseline 1：Vanilla Dense Retriever

- 只用 topic encoder
- 是所有 dual 方案的直接对照组

#### Baseline 2：BM25

- `run_bm25_baseline.py`
- 手写 BM25 实现
- 目的：提供词法基线，避免只跟 embedding 家族比较

#### Baseline 3：Cross-Encoder Reranker

- `run_cross_encoder_baseline.py`
- 先 topic 召回，再 cross-encoder 对候选重排
- 当前用的是一个 NLI 风格 cross-encoder，而不是专门 IR reranker

## 4. 当前已经得到的关键结果

### 4.1 PoC 结果

普通 embedding 在 NLI contradiction 场景下的表现：

- entailment similarity: `0.6616`
- contradiction similarity: `0.3276`
- pairwise accuracy: `0.9463`

含义：

- 普通 embedding 并不是完全分不清 entailment / contradiction；
- 但 contradiction 仍然可能有较高相似度，因此值得单独做 constraint modeling。

### 4.2 Constraint Encoder 离线提升

在 800 个 validation triplets 上：

- baseline accuracy: `94.63%`
- constraint encoder: `96.50%`
- 绝对提升：`+1.88` 个点

含义：

- Encoder B 在“区分一致 / 冲突”方面确实学到了东西。

### 4.3 CCR@3 主结果

#### MiniLM topic model，300 queries

- Vanilla CCR@3: `47.67%`
- Best dual CCR@3: `57.28%`
- 提升：`+9.61` points
- 最优参数：`alpha=0.0, tau=0.6`

#### Local Llama topic model，60 queries

- Vanilla CCR@3: `40.56%`
- Best dual CCR@3: `57.50%`
- 提升：`+16.94` points
- 最优参数：`alpha=0.0, tau=0.6`

### 4.4 分类别结果

#### MiniLM (300)

- exclusion: `54.33% -> 60.50%`（`+6.17%`）
- negation: `22.00% -> 22.00%`（`+0.00%`）
- numeric: `66.67% -> 89.33%`（`+22.67%`）

#### Local Llama (60)

- exclusion: `45.83% -> 60.42%`（`+14.58%`）
- negation: `10.53% -> 21.05%`（`+10.53%`）
- numeric: `66.67% -> 94.12%`（`+27.45%`）

### 4.5 正式多指标结果（@10）

#### Dual vs Vanilla（Local Llama topic）

- Recall@10: `0.6617 -> 0.7178`
- Recall@100: `1.0000 -> 1.0000`
- NDCG@10: `0.6533 -> 0.7230`
- CCR@10: `0.1987 -> 0.2323`
- Proxy Faithfulness: `0.1367 -> 0.2600`

增益：

- Recall@10: `+0.0561`
- NDCG@10: `+0.0697`
- CCR@10: `+0.0337`
- Proxy Faithfulness: `+0.1233`

### 4.6 baseline 对比

#### BM25

- Recall@10: `0.5456`
- NDCG@10: `0.5421`
- CCR@10: `0.0863`

#### Cross-Encoder

- Recall@10: `0.4067`
- NDCG@10: `0.3804`
- CCR@10: `0.1660`

需要注意：

- 当前 cross-encoder 不是专门 IR reranker，而是 NLI 模型；
- 所以它弱于 dual，不应被过度解读为“cross-encoder 不行”，而更应解读为“当前 baseline 选择还不够强”。

## 5. 目前最重要的实验结论

如果你要用几句话概括当前结论，可以这么说：

### 结论 1：Dual-view retrieval 确实有效

Constraint encoder 作为一个插拔式模块，能稳定提升约束遵循相关指标。

### 结论 2：最优策略偏向“强约束过滤”

从 grid search 看，最佳点通常是：

- `alpha` 很低
- `tau` 很高

说明在当前数据分布下：

> 主题相关性本身已经足够，而真正稀缺的是对“逻辑冲突文档”的拒绝能力。

### 结论 3：收益高度依赖约束类型

当前最清楚的顺序是：

```text
numeric > exclusion > negation
```

也就是说：

- numeric constraint 最容易从 constraint filtering 中获益；
- exclusion 次之；
- negation 最难。

### 结论 4：当前系统更像“检索前防线”而非完整 end-to-end 方案

项目最成熟的是 retrieval stage，而不是 generation stage。

## 6. 当前最明显的不足与风险

### 6.1 benchmark 仍然偏 synthetic

主 benchmark `constraint_benchmark_v1` 完全由模板生成。

优点：

- 可控
- 便于系统性测 negation / exclusion / numeric

缺点：

- 分布太干净；
- 真实 query 的语言变体、歧义、跨句 scope 没有覆盖；
- 容易高估方法效果。

### 6.2 negation 仍然没有解决好

尤其 MiniLM 结果里：

- negation 类几乎没有提升

这说明当前 NLI triplet + 合成模板的训练，不足以覆盖真实 negation scope。

### 6.3 cross-encoder baseline 还不够强

当前 baseline 选的是 NLI cross-encoder，而不是专门做 IR reranking 的强模型。

所以如果投稿，需要注意 baseline 的说服力边界。

### 6.4 端到端评估还是 proxy

当前 `rag_e2e_eval.py` 的规则是：

> top1 若满足约束，就视为 faithful

这不是严格的 answer-level faithfulness 评测。

### 6.5 真实分布数据不足

目前没有 300~1000 条人工标注真实 constraint queries 的评测集。

这会影响论文说服力。
